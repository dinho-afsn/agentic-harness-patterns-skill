# 上下文工程模式

## 问题

如果不加以刻意管理，上下文窗口在每一轮都会被冗余或过时的数据填满。每次调用都重建昂贵的上下文会增加延迟。预先暴露完整的能力目录会膨胀每个 prompt。委派的子 Agent 用中间状态污染父级的上下文。这些是任何跨轮次持久化并委派给子 Agent 的 Agent 所固有的。

## 四轴框架

上下文工程分解为四个关注点：

1. **选择（Select）** -- 决定什么进入上下文窗口、何时进入、以何种粒度进入。
2. **写回（Write）** -- Agent 不仅消费上下文；它还写回到持久化存储，创建学习循环。
3. **压缩（Compress）** -- 长会话耗尽窗口；截断、紧缩和快照标记可恢复预算。
4. **隔离（Isolate）** -- 委派的工作不能污染或损坏父级的上下文或文件系统。

### 关于"写回"的说明

写回轴是一个横切原则，而非独立的模式。每个轴都涉及写回：选择被缓存和失效，压缩产生被存储的摘要，隔离边界返回父级整合的结果。写回循环——自动记忆、会话提取、持久化权限、任务状态更新——正是将无状态工具调用者转变为学习系统的关键。它作为设计原则在主技能定义中涵盖，而非单独的子文件。
